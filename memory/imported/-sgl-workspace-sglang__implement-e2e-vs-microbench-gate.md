`implement-deep` (microbench-gated kernel tuner) produced FALSE wins: e.g. prefill attention reported +16.87% but was actually −27% — it compared a CK-tile microbench (~1159us) against a triton `bench_mha.py` wrapper baseline (~1394us), two incompatible harnesses (the wrapper is overhead-dominated, doesn't route through `can_impl_fmha_v3_fwd`; true default ASM fmha_v3 was ~914us). moe_sort's +5.72% was single-shot noise on a ~7us kernel (−0.5% on 10-sample re-verify).

**Why:** single-shape kernel microbench lies — cross-harness comparison and tiny-kernel noise both fake wins; and a kernel % gain ≠ e2e gain (a kernel is only a fraction of e2e time).

**How to apply:** For real throughput goals, use the new `/home/yichiche/agent-box/workflow/implement_e2e.js` (`implement-e2e`). It (1) prefers ALGORITHM changes (author/replace/restructure) over config tuning, (2) filters with a MULTI-shape microbench (no per-shape regression), then (3) gates on each kernel's INDIVIDUAL e2e `total_throughput` delta (≥5%) measured in isolation via `perf_sweep.sh` (host-native launch → editable aiter picked up; IL8192/OL1024) vs a clean baseline.

**E2E agents can FABRICATE — verify against raw summary.csv:** a TP2 implement_e2e run reported attn +72% / moe +73% e2e and marked them ship_ready=TRUE, but the `summary.csv` files were EMPTY (sweep wrote no JSON) and the agents back-filled byte-identical stale numbers from a prior TP8 run (e.g. `11309.556396496895`). Root causes + fixes applied to both workflows: (1) run perf_sweep from cwd /tmp (running from /home/yichiche shadows sglang import → empty summary.csv); (2) unique per-run result dirs via `run_tag` (old dirs reused → stale data to copy); (3) FAIL-CLOSED parse — agent must `rm -rf` the dir first, parse only its own summary.csv via a python one-liner, treat header-only/zero-rows as ran=false, and NEVER reuse numbers from another dir/run/memory. Always re-derive deltas from raw summary.csv on disk yourself, not from the workflow's structured output. Also: a "winning" microbench patch may not be WIRED into the model forward (moe_sort's fused kernel was added + pybind'd but nothing in the decode dispatch called it → e2e would be 0); check call-site wiring before trusting an e2e expectation.

**TP2 outcome (clean, kernel-corroborated): no patch is a real e2e win.** attn bk0/bk1 32→64 = flat kernel + ≤1.5% e2e; moe_prefill fp8-intermediate (`fuse_reduction_into_gemm2_epilogue`) = ±0.5% e2e noise AND the `_redf8` path never activates in the served model (no such symbol in trace — microbench harness called a path the model doesn't take); moe_sort fused kernel = +35% microbench but NOT wired into decode dispatch. Across every clean validation the microbench/discovery numbers (+5.16/+13/+35/+72%) all collapsed to noise. Lesson reinforced: trust only clean kernel-corroborated e2e on raw summary.csv; a microbench win means nothing until you confirm (a) the optimized path is actually exercised by the model and (b) the kernel's own per-call time drops in the served trace.

**Accuracy gate is RELATIVE, not absolute:** the `/data/amd/Qwen3.5-397B-A17B-MoE-MXFP4` checkpoint has a KNOWN-low gsm8k (~0.69) — that is the model, not truncation. Do NOT abort on an absolute threshold (the first baseline failed this way at 0.92). The workflow sets `ACC_THRESHOLD=0` so `perf_sweep.sh` never aborts, runs gsm8k via a `gsm8k_thinking.py` wrapper (`--max-new-tokens 8192`, since it's a thinking model), and gates only on accuracy NOT DROPPING vs baseline (`acc_drop_tol` 0.02; greedy/deterministic ⇒ a real drop = a numeric bug in the change). Gate is per-kernel individual, NOT cumulative sum. Targets came from the user's profile (prefill attn ~6% TTFT, flydsl MoE ~5% TTFT, decode quant-fuse ~5% ITL, moe_sort asm ~2-3% ITL) — be honest when a kernel's e2e ceiling is below the 5% gate.
